Architectural Approaches

How should weld's internals be structured?

A

Monolithic Single-Binary

One crate, one binary. All logic — diff engine, TUI rendering, file I/O, merge operations — lives in a single crate with module separation.

Pros

  • Simplest to start — no workspace overhead
  • Fast iteration, easy refactoring
  • Single compile target

Cons

  • Harder to extract a library later
  • Compile times grow with the whole project
  • Tighter coupling risk as it grows
B

Workspace: Core Library + TUI Binary

Cargo workspace with two crates: weld-core (diff engine, merge logic, file ops) and weld-tui (ratatui UI, keybindings, rendering). Core is a library, TUI is the binary.

Pros

  • Clean separation from day one
  • Core is testable without TUI
  • Enables future frontends (GUI, web, LSP)
  • Faster incremental compiles (change UI → only rebuild TUI)

Cons

  • Slightly more setup upfront
  • Need to think about API boundaries early
C

Workspace: Core + TUI + CLI

Three crates: weld-core (engine), weld-tui (interactive UI), and weld-cli (non-interactive commands like batch diff, exit codes for scripting).

Pros

  • Scriptable from day one
  • Maximum modularity

Cons

  • Premature — CLI can be added to TUI binary later
  • More crates to maintain before you need them

Recommendation: B

The two-crate workspace hits the sweet spot. It's barely more setup than a single crate (just a workspace Cargo.toml), but gives you testable core logic and clean boundaries from the start. Option C is premature — when you want a CLI mode, just add a --batch flag to the TUI binary.
